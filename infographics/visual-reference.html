Claims System · Visual Reference

The whole system on one page — domain model, coverage rules as data, the gates-then-math adjudication pipeline, the cost-share switch, a claim worked end to end, the accumulator's memory, the decisions that shaped the engine, the state machines, and the 27-cycle TDD build order. Every section is example-induced with real integer-cents numbers. Use it for the round-2 pairing walkthrough and to align any future agent (or human) on the model.

01 · The domain model

Eight entities, cleanly split by what they are: rules are static config, facts are what was submitted, the accumulator is running memory, and adjudications are append-only decisions. That separation is what makes adjudication deterministic and replayable.

EntityLayerOne-line responsibility
MemberidentityThe insured person who submits claims; holds PHI (name, dob).
PolicyrulesBinds a member to coverage for one plan year — active dates, deductible, OOP max.
CoverageRulerulesHow one service is covered: cost-share mechanism, deductible applicability, unit-typed limit, prior-auth.
ClaimfactsA submission envelope grouping line items; status derived, fingerprinted.
LineItemfactsOne service on a claim — the actual unit of adjudication.
AccumulatormemoryPer-member-per-plan-year running totals: deductible met, OOP met, per-service limit used.
AdjudicationdecisionsThe immutable decision for a line item (status, payable, member owes, reason, explanation). Append-only.
DisputedecisionsA member challenge that reopens a line item and preserves the prior decision.

What this proves

  • Four layers, four responsibilities. Rules (static) · facts (claim/line) · memory (accumulator) · decisions (adjudication, append-only). A decision is always reproducible from rules + facts + a snapshot of memory.
  • The decision lives on the line item, not the claim. LineItem 1—N Adjudication is what lets a re-adjudication append a new row while the original is preserved — claim status is derived, never the source of truth.
  • One policy per member per plan year (Member 1—1 Policy), and one accumulator set per plan year (Member 1—N Accumulator) — limits exhaust correctly across claims within the year.

02 · Coverage rules are data, not code

Coverage is a typed config record — not hardcoded branches, not a DSL. A discriminated cost_share union forces exactly one mechanism per service, and a unit-typed limit expresses both dollar caps and visit caps without nullable junk fields. 

type CoverageRule = {
  policy_id: string;
  service_code: string;          // closed 12-entry catalog; unlisted → NO_COVERAGE
  covered: boolean;
  excluded: boolean;             // explicit EXCLUDED beats "not covered"
  cost_share:                    // discriminated union — one mechanism per service
    | { type: "full_coverage" }
    | { type: "copay"; copay_cents: number }
    | { type: "coinsurance"; rate: number };   // member share, 0.0–1.0
  applies_deductible: boolean;   // copay → usually false; coinsurance → usually true
  limit:                         // unit-typed — the load-bearing addition
    | { unit: "none" }
    | { unit: "dollars"; amount_cents: number } // "$Y/yr" — the brief's example
    | { unit: "visits"; count: number };        // "20 PT visits/yr"
  requires_prior_auth: boolean;
};

Mini-example — the brief's "$Y/year"

Chiropractic's $1,500/yr cap is just one row's limit field:

{ service_code: "CHIROPRACTIC",
  cost_share: { type: "copay", copay_cents: 2500 },
  limit: { unit: "dollars", amount_cents: 150000 } }

What this proves

  • The discriminated union encodes a real-world fact: a benefit uses exactly one cost-share mechanism. Nullable copay_cents + coinsurance_rate fields would make the illegal "both set / neither set" states representable; the union makes them unrepresentable.
  • The unit on limit is load-bearing. A dollars-only field cannot express "20 PT visits/year" — the most common real-world cap. One discriminator covers both, and the adjudicator picks the column from it.
  • 12 rules exercise every branch: full_coverage, copay, coinsurance, visit-limit, dollar-limit, prior-auth, and excluded — plus any unlisted code falling through to NO_COVERAGE. Adding a 13th service is a data row, not a code change.

03 · The adjudication pipeline — gates, then math

Each line item walks one ordered pipeline. The first half is gates — any one fails, the line short-circuits to a clean denial. The second half is math — it always runs to completion and moves money.

Mini-example — short-circuit at the prior-auth gate
An MRI line, billed_cents: 120000, but prior_auth_present: false. Step 4 fails. The line stops immediately — steps 5–9 never run, no math, no deductible touched. Result: DENIED, PRIOR_AUTH_REQUIRED, payable 0, member owes 0, deltas all 0.

What this proves

  • The pipeline has two halves with different jobs: gates 0–5 can only deny and stop; math 6–9 can only compute and continue. No step does both.
  • A failed gate is a clean decision (HTTP 200 + reason), not an error. It short-circuits before any money moves — so a denied line is cheap and fully explainable.
  • The straddle (7b) and OOP cap (8) are the only steps that adjust rather than deny: they cap a payable and re-attribute the shortfall, leaving the line APPROVED.

04 · The cost-share switch

Step 7 is one switch on cost_share.type — exactly one mechanism per rule, never stacked. The deductible is touched by coinsurance only; copay waives it and full_coverage ignores it.

typeDeductibleMember owesPlan paysReasons
full_coverageuntouched0allowed[APPROVED]
copaywaived; accrues to OOP onlymin(copay, allowed)allowed − member[APPROVED, COPAY_APPLIED]
coinsuranceapplied first (if applies_deductible)dedPortion + coinsPortionallowed − member[APPROVED, DEDUCTIBLE_APPLIED?, COINSURANCE_APPLIED]
The coinsurance formula — the only non-trivial branch
remainingDeductible = max(0, deductible_cents − deductible_met_cents)
dedPortion   = applies_deductible ? min(remainingDeductible, allowed) : 0
remainder    = allowed − dedPortion
coinsPortion = Math.round(rate * remainder)        // round half-up
member       = dedPortion + coinsPortion
plan         = allowed − member                    // computed last → shares always sum
(a) full_coverage — $200 preventive line
allowed = 20000c. Member 0c, plan 20000c. Deductible and OOP untouched by the mechanism. [APPROVED].
(b) copay — $25 copay on a $180 visit
allowed = 18000c, copay 2500c. Member = min(2500, 18000) = 2500c; plan = 15500c. Deductible waived; 2500c accrues to OOP. [APPROVED, COPAY_APPLIED].
(c) coinsurance — 20% on a $1200 MRI, $300 deductible remaining
allowed = 120000c, remainingDeductible = 30000c. dedPortion = 30000c; remainder = 90000c; coinsPortion = round(0.20 × 90000) = 18000c; member = 48000c; plan = 120000 − 48000 = 72000c. [APPROVED, DEDUCTIBLE_APPLIED, COINSURANCE_APPLIED].

What this proves

  • One rule = one mechanism. The switch is exhaustive over a discriminated union — incoherent "copay + coinsurance" rules can't be expressed, let alone mis-adjudicated.
  • plan = allowed − member is computed last, so member + plan always equal the allowed amount — no lost or duplicated cent, even with odd-cents rounding.
  • Only coinsurance reads/draws the deductible. Copay deliberately waives it and routes the flat charge to OOP — matching how real office-visit copays behave.

06 · The accumulator — the system's memory

One table keyed by member_id + plan_year, with a dimension discriminator per row. This is what makes limits exhaust correctly across separate claims within a year, and it stores dollar limits and visit limits in different columns of the same shape.

dimensiontrackscolumn used
DEDUCTIBLEdeductible metused_cents
OOPout-of-pocket metused_cents
LIMIT:<service_code>per-service usageused_cents (dollar limit) or used_count (visit limit)

Row shape: (member_id, plan_year, dimension, used_cents, used_count). The rule's limit.unit picks the column — dollars → used_cents += plan pay; visits → used_count += 1. LIMIT: rows are created lazily on first use.

L2 — PCP_VISIT, $25 copay: copay waives the deductible and accrues to OOP only → OOP += 2500 (20000 → 22500). Deductible untouched.
L3 — MRI, 20% coinsurance, applies_deductible: remaining deductible 30000c drawn first → DEDUCTIBLE += 30000 (20000 → 50000, now met); coinsurance 18000c on the remainder → member 48000c → OOP += 48000 (22500 → 70500).

Dollar vs visit columns: had a Chiropractic line landed, it would write LIMIT:CHIROPRACTIC.used_cents += plan pay. A PT line writes LIMIT:PHYSICAL_THERAPY.used_count += 1 — same row shape, the unit decides which counter moves.

What this proves

  • The dimension discriminator avoids a column-per-service-code schema. One row shape handles deductible, OOP, and N per-service limits; a new limited service adds a lazily-created LIMIT: row, not a migration.
  • Cross-claim memory is the whole point. The before-state isn't theoretical — it's a prior claim's residue. Limits exhaust correctly because every claim reads then increments the same keyed totals.
  • One shape, two units. used_cents vs used_count keeps dollar caps and visit caps in the same table while preserving the unit's meaning — no float, no overloaded single column.

07 · Decisions that shaped the engine

Three resolved judgment calls behind the engine's behavior — each one trades a real alternative for determinism and explainability under a 48-hour, no-reviewer-queue scope.

Decision #8
Prior-auth missing = clean DENY, not NEEDS_REVIEW
A required-auth line with prior_auth_present: false becomes a clean DENIED carrying PRIOR_AUTH_REQUIRED, payable 0.
Why: there is no human-reviewer queue in scope. A NEEDS_REVIEW line would never resolve and would freeze claim aggregation at UNDER_REVIEW forever. A clean deny is deterministic and fully explainable.
Decision #12
Adjudication outcomes are DECISIONS (HTTP 200), not errors
Every denial returns 200 with status + reason + explanation. Only malformed/identity-failed input is 4xx.
InputHTTPOutcome
prior_auth_present: false200denied line, PRIOR_AUTH_REQUIRED + explanation
prior_auth_present omitted200adjudicated as auth present (default true)
billed_cents: "abc"400intake error, nothing persisted
Why: an HTTP error explains nothing to a member and would break "5 line items, 1 denied" and "explain why it was denied." A denied line is a processed line.
Decision #13
prior_auth_present defaults to true (absence = auth present)
When a line omits prior_auth_present, treat it as true. The PRIOR_AUTH_REQUIRED path fires only on an explicit false.
Why: most services don't require prior auth, so a true default keeps the common input frictionless; callers assert false only when relevant. (Documented as a demo simplification — real systems treat absence as unauthorized.)

What this proves

  • Every decision is grounded in a scope constraint (no reviewer queue, 48h budget), not preference — and each is reversible at a single seam if the constraint lifts.
  • The decision/error boundary is the engine's core contract: business denials are data (200 + reason), only bad input is a protocol error (4xx).
  • The true default makes the happy path frictionless while keeping the denial path one explicit false away — the divergence from real systems is named, not hidden.

08 · Immutability, disputes & PHI

Decisions are append-only, each line's decision-plus-accumulator-write is one transaction, and PHI is kept structurally unreachable by the engine. Together these make the system auditable, consistent, and honest about sensitive data without over-building.

(a) Adjudications are append-only

Re-adjudication never mutates the original row. It reverses the line's original accumulator deltas, reapplies the new ones, and appends a new Adjudication. History is the audit trail.

(b) One transaction per line item

The decision write and every accumulator increment (deductible / OOP / limit) commit together in a single better-sqlite3 transaction. Synchronous single-writer → no interleaving, no half-applied decision.

(c) PHI is structurally unreachable

Member.name/dob and Claim.provider/diagnosis_code are sensitive. The engine input type keys on member_id only — it cannot read PHI because those fields aren't in its input shape. Column-level encryption-at-rest is documented as a candidate, not over-built for a 24–48h take-home.

Mini-example — disputing a denied MRI line

The original DENIED row is preserved verbatim; the new APPROVED row is appended with a DISPUTED_OVERRIDE reason, and the line's accumulator deltas are reversed then reapplied — both writes in one transaction.

What this proves

  • Append-only is the audit story. You can always answer "what did we decide, when, and why" — the original decision survives every dispute, so there's no destructive overwrite to reconstruct.
  • Atomicity is per line item. Decision + accumulator increments are one transaction, so the accumulator can never drift from the decisions that produced it — even mid-claim.
  • PHI minimization is enforced by the type system, not by discipline. The engine literally has no field to read diagnosis_code from; encryption-at-rest is named as the next step rather than half-implemented.

09 · State machines & aggregation

The engine produces the line-item state directly; the claim state is derived by aggregating its lines — never set directly. NEEDS_REVIEW is now reachable only through a dispute reopen.

Line-item lifecycle
Line outcomesDerived claim status

every line APPROVED (full payable)APPROVED
mix of APPROVED + DENIED, or any dollar-straddle partialPARTIALLY_APPROVED
any line NEEDS_REVIEW / disputed re-adjudicatingUNDER_REVIEW
every payable line PAID, none openPAID
Mini-example: the worked claim from §05 — lines [DENIED, APPROVED, APPROVED] → a mix → claim PARTIALLY_APPROVED. Flip L2/L3 to disputed and re-adjudicating → claim UNDER_REVIEW.

What this proves

  • Claim state is derived, not stored: claimStatusFrom(lines) is a pure aggregation, killing the "claim says approved but lines say denied" class of bug.
  • PARTIALLY_APPROVED is claim-level only — never a line state. A straddled line is itself APPROVED with a LIMIT_EXCEEDED note.
  • NEEDS_REVIEW survives as a state but is now reached only via a dispute reopen — prior-auth-missing is a clean DENIED (decision #8), so it never freezes a fresh claim.

10 · The build order — 27 test-first cycles

The engine is test-driven one behavior per red→green cycle, in this order. Cycles 1–25 are pure (a plain object, no DB); only 26–27 touch SQLite. This is the coding checklist.

Gates · cycles 1–7 — pure denials
  1. full_coverage line → member 0, plan = billed
  2. no rule → NO_COVERAGE, payable 0, DENIED
  3. excluded rule → EXCLUDED, payable 0
  4. covered == false → NO_COVERAGE, payable 0
  5. service date outside policy window → POLICY_NOT_ACTIVE
  6. requires_prior_auth && prior_auth_present == false → PRIOR_AUTH_REQUIRED, clean DENY
  7. duplicate fingerprint → DUPLICATE_LINE_ITEM, payable 0, original untouched
Cost-share · cycles 8–13 — single line, fresh accumulators
  1. copay, allowed > copay → member = copay, plan = allowed − copay, deductible untouched
  2. copay, allowed < copay → member = allowed, plan = 0 (min clamp)
  3. coinsurance, deductible already met → member = round(rate·allowed), plan = rest
  4. coinsurance, allowed < remaining deductible → all to deductible, coins 0, plan 0
  5. coinsurance deductible crossing → split ded + coins; shares sum to allowed
  6. coinsurance odd cents (20% of 3333) → deterministic rounding; member + plan == allowed
Limits · cycles 14–17
  1. visits, used < count → approved, limit_inc = 1
  2. visits, used ≥ count → LIMIT_EXCEEDED, payable 0 (whole-visit, no straddle)
  3. dollars, payable within remaining → approved, limit_inc = plan pay
  4. dollars straddle → plan caps at remaining, shortfall→member, LIMIT_EXCEEDED, line APPROVED
OOP cap · cycles 18–19
  1. member share would pass oop_max → excess refunded to plan, OOP_MAX_REACHED
  2. accumulator already at oop_max → next line plan pays 100%, member 0
Cross-line / determinism · cycles 20–21
  1. two coinsurance lines, one claim → line 2 sees line 1's advanced deductible
  2. re-run identical claim on identical snapshot → identical results
Aggregation · cycles 22–25 — line[] → claim
  1. all approved → claim APPROVED
  2. all denied → claim DENIED
  3. mix approved + denied → claim PARTIALLY_APPROVED
  4. straddled-partial + full approved → claim PARTIALLY_APPROVED
Writeback / integration · cycles 26–27 — SQLite
  1. full claim through the real store: accumulators persist, adjudications append-only, one txn
  2. dispute reopens a line, re-adjudicates, preserves the original immutably (last)

What this proves

  • The order is gates → cost-share → limits → OOP → cross-line → aggregation → persistence: each cycle adds exactly one behavior on top of green, so a failing test always names the behavior that broke.
  • Cycles 1–25 need no database — they run against a plain accumulator snapshot object. The pure core is fully proven before SQLite ever appears.
  • Determinism (cycle 21) and immutable disputes (cycle 27) are explicit tests, not hopes: re-running a claim on the same snapshot must reproduce identical cents.
