Claims System · Visual Reference

Four diagrams that explain the whole system: money flow, domain model, adjudication pipeline, and state machines. Use this for pairing-round walkthroughs and to align any future agent (or human) on the model.

01 · The money flow

A claim is a request for reimbursement. Adjudication splits the billed amount across member-owed slices (deductible, copay, coinsurance) and the insurer's payable share. The order matters: change it and every downstream dollar is wrong.

What this proves

  • Adjudication is deterministic. Same inputs → same outputs, every time. This is testable.
  • Money is stored as integer cents. Float arithmetic on money is the most common claims bug in industry.
  • The EOB is not a separate report — it falls out of the adjudication's reason codes. One source of truth.

02 · The domain model

Two bounded contexts. The adjudication context is pure functions, no I/O. The submission context handles the wire envelope. The seam between them is the only place persistence happens. This separation is what keeps the engine testable.

What this proves

  • CoverageRule is data, not code. A typed config object — not a DSL — is the 80/20 within 48h.
  • Accumulator is the system's short-term memory. Reading it before adjudication and writing it after is the only transactional step the engine has.
  • The Claim → LineItem 1:N relationship is why partial approvals exist. The system grades each line independently and rolls up.
  • Dispute doesn't mutate the original adjudication — it reopens a line item and produces a new one with a DISPUTED_OVERRIDE reason. History is preserved.

03 · The 11-step adjudication pipeline

Each line item runs through these 11 steps in this exact order. Steps 1-5 can short-circuit with a denial reason. Steps 6-11 do the arithmetic and always run to completion. Each step is a pure function. Each reason code has at least one test.

What this proves

  • The pipeline is composable pure functions, not a single mega-function. Each step takes (state, line_item, policy) and returns either continuation or short-circuit.
  • A line item can carry multiple reason codes. A typical approved claim carries DEDUCTIBLE_APPLIED + COPAY_APPLIED + COINSURANCE_APPLIED simultaneously.
  • The writeAccumulator step happens exactly once, at the end. No partial writes, no race conditions within a single adjudication.
  • Adding a new reason code is localized: one new step (or one branch in an existing step), one explanation template, one test. The shape of the system absorbs the change.

04 · State machines & aggregation

Two state machines move independently. The line-item machine is what the engine produces directly. The claim machine is derived from line-item states via the aggregation function — it is never set directly.

What this proves

  • Claim state is derived, not stored. claimStateFrom(items) is a pure function. This eliminates an entire class of "claim says approved but line items say denied" bugs.
  • The aggregation table has exhaustive test coverage: each row is one test case. NEEDS_REVIEW overrides everything else, which is the rule a real claims adjuster would expect.
  • The dispute loop reopens any terminal state. From a state-machine perspective, terminal isn't actually terminal — it's "terminal-until-disputed."
  • The PAID transition is the only one that requires a side effect (recording payment). Everything else is in-memory state arithmetic.
